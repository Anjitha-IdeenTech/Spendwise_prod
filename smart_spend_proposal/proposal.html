Section 02: Product Philosophy Smart Spend Request

A Paradigm Shift in Philosophy

We are changing the fundamental operational philosophy. Instead of forcing users to understand backend procurement systems, we build a layer that does the thinking for them.

Current Philosophy
"Users must understand procurement"

Employees undergo hours of ERP training, navigate hundreds of database screens, and learn procurement rules just to order basic office hardware.

Proposed Philosophy
"Users should not need to understand procurement"

Employees request what they need in natural language. An intelligent, automated backend handles budget tracking, contract matching, and Odoo routing.

Key Shift: The "Consumer-Like" Experience

By introducing the Smart Spend Request Platform, we decouple the transacting user from the governing workflow. The end-user experiences a simple portal akin to requesting an item on Amazon, while the procurement team retains absolute control, audit logs, and ERP compliance via Django and Odoo behind the scenes.

Section 05: Backend Architecture Smart Spend Request

Backend Workflow & Governance

Behind the simplified user portal, our business logic engine maintains strict corporate governance. If contracts exist, POs are created automatically; if not, sourcing begins.

Process Stage Flow A: Rate Contract Active (Fast Path) Flow B: No Contract Active (Sourcing & CR Path)
1. PR Initiation Portal auto-matches active Rate Contract; pre-fills contracted prices, vendor, and terms. Portal detects no active contract. Marks request as "New Sourcing" for buyer action.
2. PR Approval Routes immediately through standard amount-banded approvals (Workflow: pr). Triggers and routes background "Need for Contract" approvals (Workflow: need_cr).
3. Sourcing (CR) Bypassed. Directly proceeds to order generation without manual buyer intervention. Creates **Contract Request (CR)**. Purchase Head assigns Buyer to run RFQ, Bidding, or Negotiation.
4. Contract & PO Final approval auto-creates, groups, and confirms Purchase Order(s) in Odoo. Bids compared; contract approved & signed. Portal auto-triggers Odoo PO from contract.

Reassurance for Management: Governance is Maintained

This workflow proves that the platform does not bypass rules. If a contract is available, the backend speeds execution by automating the PO. If no contract exists, it automatically falls back to full procurement sourcing (RFQ, bidding, negotiation) in Odoo, ensuring that budget rules, compliance, and buyer assignment remain robust.

Section 09: Technology Architecture Smart Spend Request

Technology Architecture

Our multi-tier architecture links a modern Next.js client interface with a Django orchestrator, using Odoo's native XML-RPC APIs to manage compliance and data sync.

Seamless Frontend: Next.js delivers a responsive client dashboard. It manages active states, renders interactive forms, and enables instant page transitions.
ERP Integrity: All data resides securely in the Odoo database. The Django engine triggers standard Odoo processes via XML-RPC APIs, ensuring data consistency and security.
